# Skill 工匠框架

> 审计和构建高质量 AI Skill 的 10 维度评估体系。不是"能跑就行"——每个维度三级标准（★★★/★★☆/★☆☆），条件判断框架，不是固定清单。

## 这是谁的工具？

为 **Skill 作者和 AI 工作流设计者** 设计。如果你：
- 写了不少 Skill，想知道哪个在拖后腿、怎么修
- 正在设计新 Skill，不想踩已知的坑
- 需要一套可复用的质量标准来 review 团队的 Skill

那这是你的工具。如果你只用一个 Skill、且它是从别人那抄的、也不打算改 → 不需要这个。

## 30 秒跑起来

```
克隆到 Claude Code skills 目录。
触发词：检查这个 skill / 评估这个工作流 / 按工匠标准 / skill quality audit
```

两种模式：
- **模式 A（审计）**：给一个已有 Skill → 10 维度逐项打分 → 输出最弱 2 维度的具体改进建议
- **模式 B（构建）**：从零设计新 Skill → 10 维度逐步指导 → 每步红绿灯检查

Phase 0 自检门禁：框架先审计自己，self-audit.md 过期/不存在 → 硬阻断，拒绝执行。

## 10 维度框架

| # | 维度 | ★★★ 标准 |
|---|------|---------|
| 1 | 触发精准度 | 精确触发 + 互斥声明 + 负面测试 + 模糊入口 |
| 2 | 信息架构 | L1→L2→L3 三层，Body < 500 行，按需加载 |
| 3 | 验证体系 | 独立验证 + 可操作通过标准 + 失败信号 + 迭代上限 |
| 4 | 用户视角设计 | 一个意图一个入口，参数区分行为 |
| 5 | 诚实边界 | 3+ 具体局限 + 截止时间 + 不适用场景 + 承认推测 |
| 6 | 产物自包含 | 单目录复制即用，零外部依赖 |
| 7 | 表达一致性 | 可验证的风格规则 + 每条有反例 |
| 8 | 迭代纪律 | 迭代上限 + 终止条件 + 每轮聚焦维度 |
| 9 | 交付物链 | 每步声明输入/输出文件，下游验证上游，缺失→硬阻断 |
| 10 | 层层外化门禁 | 门禁文件化 + PASS/FAIL 判词 + 下游检查 + 人类可读 |

## 6 个决策分支

3 位 Skill 作者（花叔·Zara Zhang·Anthropic）的方法论在 6 个点上存在分歧。不强行统一——每个分歧是条件判断分支，不是唯一正确答案。

| 分支 | 条件 A → 策略 | 条件 B → 策略 |
|------|-------------|-------------|
| 审批策略 | 错误成本高，返工代价大 → 设检查点 | 产物固定，质量波动有上限 → 直接建 |
| 输出标准 | 产物类型单一 → 固定清单 | 产物类型多样 → 原则性指导 |
| 优化方式 | 可量化（触发率/token） → 自动化 | 主观质量（表达DNA/品味） → 人工精炼 |
| 矛盾处理 | 评估"人/思维" → 保留矛盾 | 评估"功能/流程" → 要求一致性 |
| 先后顺序 | 产物定义清晰 → Build First | 产物在探索中 → Eval First |
| 流程保证 | ≥3 串行步骤 → 结构保证（文件链） | ≤2 步 → 约定保证 |

## 11 条红线

3 位作者一致认定不可违反：

1. 不编造（AI 推测包装成事实）
2. 不把通用道理包装成独特见解
3. 不忽略负面/批评视角
4. 不自评（自己写自己验证）
5. 不修改原始数据
6. 不用不可验证的风格指令
7. 不无限打磨（2 轮后标注诚实边界，交付）
8. 不依赖外部环境
9. 不设无文件化的门禁
10. 不设无输入验证的步骤
11. 结构图不同步（改源文件不更新 system-diagram.html）

## 设计哲学

**先审自己，再审别人。** Phase 0 自检门控是框架最独特的设计——在审计任何 skill 之前，框架先检查自己的 self-audit.md 是否过期。如果连自己都审不了，凭什么审别人？这建立了一种非同寻常的信任。

**情境 > 教条。** 三位作者的 6 个决策分支不是"谁对谁错"的争论，而是"什么时候听谁的"的情境判断。构建优先 vs 评估优先、检查点 vs 直接构建——取决于错误成本和输出类型。框架不强行统一方法论，承认分歧是真实的、取决于情境的。

**交付物链是 Agent 防跳步机制。** 维度 9 来自实战教训——SOP 步骤被 Agent 跳过而系统未发现。文件链修复了这个漏洞：每一步声明输入/输出文件，下游验证上游文件存在，缺失即硬阻断。对话中的门控不算门控——必须写磁盘。

**反例让原则可操作。** 每个维度的评分标准都包含反例——不是"你应该做 X"，而是"做 Y 是错误的，因为……"。反例来自三位作者的实际踩坑，不是理论推演。

## 表达风格规则（框架自身遵循）

| # | 规则 | 反例 |
|---|------|------|
| 1 | 证据必须引用原文行号 | ~~"描述不太够"~~ → "第 54 行只覆盖 2 种触发场景" |
| 2 | ★★☆ 及以下评分必须给出改后文本示例 | ~~"需要改进触发描述"~~ → 给出具体改写 |
| 3 | 证据强度必须标注 | 实战暴露 / 逻辑推演 / 推测 |
| 4 | 改进建议必须是可操作项 | ~~"需要优化信息架构"~~ → "将 X 拆分为独立文件" |
| 5 | 不写"好问题""当然可以""综上所述" | 直接从观点切入，在最后一个观点处停止 |

## 诚实边界

- **调研截止 2026-05**。3 位作者在此日期之后的新方法论迭代未纳入
- **样本量有限**：提炼自 3 位 Skill 作者。3 个样本提供正交视角，但不能保证覆盖所有有效的 Skill 设计哲学
- **未经验证**：尚未经过大规模独立审计验证。当前是基于方法论提取和交叉比对的理论框架
- **人物 Skill 评估可能偏保守**：花叔的高标准对轻度人物复现可能过严
- **不适用场景**：极简 Skill（< 50 行单一功能）、重度依赖外部 API 的 Skill、实验性/探索性 Skill
- **已知结构性盲区**：跨维度交互盲、上下文预算盲、时间衰减盲、过度工程化盲、用户适配盲（详见 SKILL.md 诚实边界）
- **领域知识获取路径**：领域透镜 / 双轨审计 / 失败模式库
- **自审计记录**：见 `references/self-audit.md`
