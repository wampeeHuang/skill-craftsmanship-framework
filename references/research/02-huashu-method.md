> 来源：[女娲·Skill造人术（nuwa-skill）](https://github.com/alchaincyf/nuwa-skill) — SKILL.md、README.md、references/extraction-framework.md，以及花叔 CLAUDE.md（接口设计原则）

## 1. 核心设计哲学

花叔的 Skill 设计观围绕一个核心命题展开：**Skill 不是角色扮演，是认知操作系统的提取。** 以下 5 条哲学贯穿整个女娲流水线。

### 1.1 蒸馏的是 HOW they think，不是 WHAT they said

> 来源：SKILL.md 第 25 行

**具体含义**：Skill 不应该复读名人名言，而应该捕捉一个人的认知框架——他用什么心智模型看世界、用什么决策启发式做判断、面对新问题时如何推导。花叔在 README 的效果示例中特别强调：乔布斯的回答"不是在复读名人语录，是在用名人的认知框架帮你分析"。

**可操作标准**：
- 每个心智模型必须通过三重验证（跨域复现 + 生成力 + 排他性）
- Skill 生成后必须做边缘测试（Edge Case）：用此人没讨论过的问题测试，看 Skill 能否基于模型推导，而非拼凑原话
- "不是原话的拼凑，而是框架的运行"（extraction-framework.md 质量自检项）

### 1.2 宁可 60 分诚实，不要 90 分编造

> 来源：SKILL.md 第 312 行

**原话**："宁可生成一个诚实标注了局限的60分Skill，也不要生成一个看起来完美但实际上在编造的90分Skill。"

**具体含义**：信息不足是常态，编造是不可接受的。当某个维度公开信息很少时，宁可缩短心智模型数量（减至2-3个）、加大诚实边界篇幅，也不强行生成看起来完整的 Skill。

**可操作标准**：
- 冷门人物（可用来源 < 10 条）：在 Phase 0.5 就提醒用户，心智模型减至 2-3 个，每个都标注"基于有限信息推测"
- 诚实边界至少 3 条具体局限，不能只有"不能替代本人"这种空话
- "一个不告诉你局限在哪的 Skill，不值得信任"（README.md）

### 1.3 调研质量决定 Skill 上限，垃圾进垃圾出

> 来源：SKILL.md 第 337 行

**原话**："调研质量决定了最终Skill的上限。垃圾进垃圾出，在这里拦截比在Phase 4返工成本低得多。"

**具体含义**：花叔把 6 个 Agent 的并行采集放在流程最前端，并设计了 Phase 1.5 检查点来确保调研质量。这是"检查点思维"在流水线设计中的体现——在关键节点设卡，而不是到最后才发现问题。

**可操作标准**：
- 每个 Agent 调研结果必须写入独立文件（`references/research/0X-xxx.md`），"不存文件的调研等于没做"
- 每条信息标注来源 URL 和可信度（一手 > 二手 > 推测）
- 区分"他说过的" vs "别人说他的" vs "我推断的"

### 1.4 Skill 必须是自包含的

> 来源：SKILL.md 第 169 行

**原话**："所有调研文件必须存在skill目录内部（`references/research/`），绝对不要存到其他外部目录。Skill必须是自包含的——复制整个skill目录就能独立使用，不依赖任何外部文件。这是为开源分发设计的核心原则。"

**具体含义**：每个 Skill 是一个完整的、独立的单元。不依赖项目根目录的其他文件，不依赖外部路径。复制即用。

**可操作标准**：
- 目录结构固定：`SKILL.md` + `scripts/` + `references/research/` + `references/sources/`
- `scripts/` 放工具脚本（字幕下载、清洗、质量检查）
- `sources/` 放一手素材（书籍、transcript、文章）

### 1.5 矛盾是深度来源，不是需要修复的 Bug

> 来源：extraction-framework.md "矛盾处理原则"

**原话**："矛盾是人格的核心特征，不是需要修复的Bug。""这恰恰是深度的来源。"

**具体含义**：不试图调和矛盾、不选边站、不假装矛盾不存在。三种矛盾类型各有处理方式：
- 时间性矛盾（观点演化）：记录演化轨迹，以近期为主但提及变化
- 领域性矛盾（不同场景不同规则）：分领域记录，不强求统一
- 本质性张力（价值观内在冲突）：明确记录为"核心张力"

**可操作标准**：Phase 4 通过标准明确要求"内在张力：至少 2 对矛盾"，观点高度一致反而是不通过信号（"太假"）。

## 2. 流程设计原则

花叔的 7 阶段流水线（Phase 0-5）体现了 8 条可复用的流程设计原则。

### 2.1 Phase 0 双入口分流：明确人名 vs 模糊需求

> 来源：SKILL.md Phase 0（第 30-38 行）

**核心机制**：同一个 Skill 支持两种使用路径，通过意图分流而非多 Skill 拆分实现。

| 用户输入 | 路径 | 说明 |
|---------|------|------|
| 明确的人名/主题 | Phase 0A 直接路径 | 确认细节后直接蒸馏 |
| 模糊的需求/困惑 | Phase 0B 诊断路径 | 从需求反推最合适的蒸馏对象 |

**设计启示**：不要因为用户意图不同就建立多个 Skill。用入口分流在同一个 Skill 内处理不同路径，减少用户的选择成本和 Skill 碎片化。

### 2.2 6 Agent 并行采集（Agent Swarm）

> 来源：SKILL.md Phase 1（第 173-226 行）

**核心机制**：启动 6 个并行 subagent，每个负责不同的信息维度：

| Agent | 搜索目标 | 提取重点 |
|-------|---------|---------|
| 1 著作 | 书、长文、论文、newsletter | 反复出现的核心论点（≥3次）、自创术语、推荐书单 |
| 2 对话 | 播客、长视频、AMA、深度采访 | 被追问时的回答方式、即兴类比、改变立场的瞬间 |
| 3 表达 | Twitter/X、微博、即刻、短文 | 高频用词句式、争议立场、幽默方式 |
| 4 他者 | 他人分析、书评、批评、传记 | 外部观察到的模式、批评与争议 |
| 5 决策 | 重大决策、转折点 | 决策背景与逻辑、事后反思 |
| 6 时间线 | 出生到现在的完整时间线 | 关键里程碑、思想转折点、最近12个月动态 |

**设计启示**：
- 按信息维度（不是按数据源）分配 Agent——每个 Agent 有明确的"提取重点"，而不是"去搜 Google"
- Agent prompt 模板包含：任务目标 + 具体搜索方向 + 输出格式要求 + 信息源黑名单
- Agent 之间可能有冲突结果——保留矛盾，用"内在张力" section 收录

### 2.3 三重验证筛选：跨域复现 > 生成力 > 排他性

> 来源：SKILL.md Phase 2.1（第 347-356 行）+ extraction-framework.md（第 6-33 行）

**核心机制**：从 15-30 个候选论点中筛选 3-7 个心智模型，三重门槛：

| 验证 | 标准 | 判定 |
|------|------|------|
| 跨域复现 | 在 ≥2 个不同领域/话题中出现 | 排除随口一说的观点 |
| 生成力 | 能推断此人对新问题的可能立场 | 排除不能迁移的特定场景观点 |
| 排他性 | 不是所有聪明人都会这样想 | 保留此人的独特视角 |

**三重通过** → 心智模型；仅 1-2 重 → 降级为决策启发式；0 重 → 丢弃

**设计启示**：这是花叔方法论的"塔勒布时刻"——用排他性作为取舍标准，避免 Skill 变成通用鸡汤。排他性越强越靠前排序。

### 2.4 检查点思维：在关键节点设卡，不在最后才发现问题

> 来源：SKILL.md Phase 1.5（第 314-337 行）和 Phase 2.5（第 393-409 行）

花叔在流水线中插入两个强制检查点，每个都需要用户确认后才能继续：

**Phase 1.5 调研 Review 检查点**：
- 6 个 Agent 完成后暂停
- 展示调研质量摘要表格（来源数量 + 关键发现 + 矛盾点 + 信息不足维度）
- 用户确认调研质量 OK → 进入 Phase 2
- "在这里拦截比在 Phase 4 返工成本低得多"

**Phase 2.5 提炼确认检查点**：
- Phase 2 提炼完成后暂停
- 展示提炼摘要（心智模型名称列表 + 决策启发式数量 + 表达 DNA 关键词 + 核心张力 + 诚实边界）
- 用户确认 OK → 进入 Phase 3 构建
- "提炼是主观判断最重的环节，确认后再构建，避免写完 400 行 SKILL.md 才发现方向不对"

### 2.5 Agentic Protocol：让 Skill 不只"说得像"还"做得像"

> 来源：SKILL.md Phase 3 Step 2 "回答工作流生成指引"（第 439-495 行）

**核心创新**：每个人物 Skill 包含一个 3-Step 回答工作流，其中 Step 2 的研究维度**从蒸馏出的心智模型自动推导**，不是固定模板。

| 人物 | 核心心智模型 | → 推导出的研究维度 |
|------|------------|------------------|
| 芒格 | 多元思维模型、逆向思考、激励机制 | → 看护城河、看管理层激励结构、看最大风险（逆向）、看历史类比 |
| 费曼 | 第一性原理、对权威的怀疑 | → 看基本物理/数学约束、看官方说法的逻辑漏洞、看实验数据 |
| 塔勒布 | 反脆弱、尾部风险、知识的僭妄 | → 看极端情况、看谁在承担尾部风险、看专家预测的历史记录 |

**关键约束**：研究维度必须来自心智模型，不能是通用的"搜索相关信息"；每个维度要有具体的搜索指引（搜什么、看什么数据）。

### 2.6 迭代上限 2 轮：不过度打磨

> 来源：SKILL.md Phase 4.4（第 537 行）

**原话**："Phase 2→4最多循环2次。如果2轮后仍有不通过项，在诚实边界中标注薄弱维度，交付当前最优版本而非无限打磨。"

**设计启示**：质量验证不是为了让 Skill 完美，而是为了拦截致命缺陷。2 轮后仍然薄弱 = 信息源本身不够好，继续打磨是在掩盖而非解决问题。

### 2.7 Phase 5 双 Agent 精炼：独立评审，交叉验证

> 来源：SKILL.md Phase 5（第 543-563 行）

**核心机制**：Phase 4 验证通过后，自动启动两个并行 Agent 从不同视角评审 Skill：

| Agent | 视角 | 输出 |
|-------|------|------|
| Agent A（auto-skill-optimizer） | 8 维度结构评估 + 3 个典型 test prompt 干跑 | 最弱 2 个维度的改进建议（含改后文本示例） |
| Agent B（skill-creator） | 触发条件覆盖 + 角色扮演规则可操作性 + 缺失信息 | 2-3 处具体文本改动建议（含改后文本示例） |

主 Agent 综合两份报告，应用不冲突的改进。"改动必须让skill'激活即执行'，不只是增加内容，而是让AI拿到skill后知道先做什么、碰到什么停下来。"

### 2.8 更新模式：增量更新，不全量重建

> 来源：SKILL.md "更新已有 Skill"（第 567-577 行）

当用户说"更新 XX 的 skill"时：
- 只启动 Agent 2（最新对话）+ Agent 5（最新决策）+ Agent 6（时间线更新）
- 对比新信息与现有内容：强化 → 补充案例；矛盾 → 标注变化并更新模型；新模式 → 考虑增加新模型
- "不重写整个 Skill，只增量更新"

## 3. 信息源管控

### 3.1 三级信息源优先级

> 来源：SKILL.md 第 288-296 行

| 优先级 | 来源类型 | 揭示什么 |
|--------|---------|---------|
| **最高+** | 用户提供的一手素材 | 完整原文，未经二手过滤 |
| **最高** | 本人著作 | 系统性思考 |
| **最高** | 长对话/访谈 | 即兴思维过程 |
| **最高** | 实际决策记录 | 真实行为 vs 声称 |
| **中等** | 社交媒体 | 表达风格、即时反应 |
| **中等** | 他人评价 | 外部视角、盲点 |
| **低** | 二手转述 | 参考但需验证 |

**核心逻辑**：权重不以"来源名气"为准，而以"揭示思维结构的直接性"为准。一手 > 二手，长文 > 碎片，行为 > 声称。

### 3.2 信息源黑名单（永远排除）

> 来源：SKILL.md 第 298-303 行

| 黑名单来源 | 排除原因 |
|-----------|---------|
| **知乎** | 洗稿严重、信息失真率高，不作为任何维度的来源 |
| **微信公众号** | 封闭生态、无法验证、大量二手转述 |
| **百度百科/百度知道** | 信息陈旧且不可靠 |

**中国人物替代方案**（Phase 0.5 完成检查项）：B站原始视频、小宇宙播客、权威中文媒体（36氪、极客公园、晚点LatePost、财新、第一财经、虎嗅、少数派、机器之心）。

### 3.3 本地语料模式的采集策略

> 来源：SKILL.md 第 175-204 行

当用户提供本地素材（PDF/transcript/字幕/文章）时，采集策略从"全网搜索"切换为"本地优先 + 缺口补充"：

1. 先读本地素材，按 6 个维度分类
2. 识别信息缺口：哪些维度覆盖充分？哪些缺失？
3. 只对缺失维度启动网络搜索 Agent
4. 调研文件中明确区分"来自用户提供素材" vs "来自网络搜索"

**设计启示**：本地一手素材权重最高——"用户手上的一手素材（尤其是完整书籍、长访谈原文）通常比网络搜索到的二手转述质量高得多。"

## 4. 质量验证体系

### 4.1 Phase 4 通过标准（6 项硬指标）

> 来源：SKILL.md Phase 4.4（第 527-534 行）

| 检查项 | 通过标准 | 不通过信号 |
|--------|---------|-----------|
| 心智模型数量 | 3-7 个，每个有来源证据 | < 3 或 > 10 |
| 每个模型的局限性 | 明确写出失效条件 | 只写优点 |
| 表达 DNA 辨识度 | 读 100 字能认出是谁 | 像通用 ChatGPT |
| 诚实边界 | 至少 3 条具体局限 | 只有"不能替代本人" |
| 内在张力 | 至少 2 对矛盾 | 观点高度一致（太假） |
| 一手来源占比 | > 50% | 主要依赖二手转述 |

**设计启示**：通过标准不只是"合格线"，还是设计目标的量化表达。每个标准都有明确的 PASS/FAIL 判定——没有灰色地带。

### 4.2 Phase 4 三测试

> 来源：SKILL.md Phase 4.1-4.3（第 506-523 行）

| 测试 | 方法 | 期望结果 |
|------|------|---------|
| **已知测试**（Sanity Check） | 选 3 个此人公开表态过的问题，spawn 子 agent 带着新 Skill 回答 | 方向一致 |
| **边缘测试**（Edge Case） | 选 1 个此人没公开讨论过但相关的问题 | "基于模型 X 和 Y 的推断，可能...但不确定"（不应斩钉截铁） |
| **风格测试**（Voice Check） | 用 Skill 写 100 字分析 | 有此人表达特征，不是通用 AI 味鸡汤，不是原话拼凑 |

**关键设计决策**：用子 agent 执行测试而非主 agent 自评——"独立于主 agent，避免自评偏差。"

### 4.3 Phase 5 双 Agent 精炼

> 来源：SKILL.md Phase 5（第 543-563 行）

Phase 4 通过后自动启动，两个并行 Agent 从不同视角做最后打磨（详见 2.7）。精炼标准："让 skill'激活即执行'，不只是增加内容，而是让 AI 拿到 skill 后知道先做什么、碰到什么停下来。"

### 4.4 质量自检清单（extraction-framework.md）

> 来源：extraction-framework.md 第六节（第 126-151 行）

分 5 个维度各自检：
- 心智模型：每个模型 ≥ 2 个不同领域证据、数量 3-7、有场景和局限、有张力但不矛盾
- 表达 DNA：有辨识度但不 caricature、核心特征而非表面模仿
- 决策启发式：每条有案例支撑、可被新情况触发
- 诚实边界：明确写做不到什么、标注来源和时间、承认不足
- 整体：用此人眼睛看新问题能得到价值、非原话拼凑、删掉名字还能认出是谁

## 5. 接口设计原则

> 来源：花叔 CLAUDE.md 第 39-43 行 + 在女娲 Skill 中的体现

### 5.1 "一个意图一个入口"

**原话**："以 agent 的意图为拆分单位，不以内部实现为拆分单位。"

**具体规则**：
- 不同行为用参数区分，不新开入口
- REST 式 CRUD 细粒度拆分（list/get/create/update/delete 各自独立）是反模式——增加 token 开销和选择错误率
- 遇到碎片化接口：合并，不继续遵循
- 设计新接口时从调用者视角出发，不从实现者视角出发
- 适用范围：工具、Skill、MCP、API、CLI、文件接口——凡 agent 消费的东西都照此原则

### 5.2 在女娲 Skill 中的体现

**体现 1**：两种入口（明确人名 + 模糊需求）共用一个 Skill，通过 Phase 0 分流，而不是创建两个 Skill（"蒸馏人名" + "诊断推荐"）。

**体现 2**：触发词覆盖多种表达方式——"造 skill""蒸馏 XX""女娲""造人""XX 的思维方式""做个 XX 视角""更新 XX 的 skill"，甚至模糊需求如"我想提升决策质量"也能触发——但都是一个入口，不是多个 Skill。

**体现 3**：更新已有 Skill 不走新流程——在同一个 Skill 内，通过"更新 XX 的 skill"触发增量更新分支，而不是建立"女娲-创建"和"女娲-更新"两个 Skill。

### 5.3 推论：从接口设计到 Skill 设计

如果将 CLAUDE.md 的接口原则推广到 Skill 设计：

- **不要在 Skill 外部区分入口**：不要在 description 中写死"只有输入人名才触发"——让 Skill 自己判断用户意图
- **用参数而非新入口做分支**：人物 vs 主题、新建 vs 更新、全面 vs 聚焦——都是同一个 Skill 内的分支，不是多个 Skill
- **从调用者视角设计触发词**：覆盖用户可能说的一切表达，而不是从实现者视角设计"正确的调用方式"

## 6. 反模式 / 绝不做的事

> 来源：SKILL.md "品味守则 > 绝不做的事"（第 591-594 行），以及全文散见的多处红线

### 四条红线

| # | 红线 | 来源位置 | 反例 |
|---|------|---------|------|

## 7. 主题 Skill vs 人物 Skill 的变体策略

> 来源：SKILL.md "特殊场景 > 主题Skill vs 人物Skill"（第 605-618 行）+ extraction-framework.md 第五节（第 113-120 行）

花叔的方法论在"人物 Skill"和"主题 Skill"两种场景下有系统性的变体策略，不是简单复用同一套模板。

### 各 Phase 对比

| Phase | 人物 Skill | 主题 Skill 变体 |
|-------|----------|--------------|
| **Phase 0A** | 确认人名 + 聚焦方向 | 确认主题边界 + 目标受众（"价值投资"是格雷厄姆式还是全流派？） |
| **Phase 0.5** | 目录名 `[person]-perspective/` | 目录名 `[topic]-framework/`，结构相同 |
| **Phase 1** | 6 Agent 围绕一个人 | 先搜该主题的 3-5 个核心人物/流派，再按人物分配 Agent（每人 1-2 个维度） |
| **Phase 2.1** | 提取一个人的心智模型 | 提取**领域共识框架**（所有流派认同的）+ **各家分歧**（A 说 X，B 说 Y） |
| **Phase 2.3** | 模拟一个人的表达 | 不模拟特定人物语气，用中性但专业的表达 |
| **Phase 2.4** | 一个人的内在矛盾 | 流派间的根本分歧（如价值投资 vs 成长投资的哲学差异） |
| **Phase 3** | 用 skill-template.md | 调整模板：去掉角色扮演规则和身份卡，改为"框架概览" + "流派对比" |
| **Phase 4** | 对比此人已知立场 | 对比领域内公认的经典案例 |

### 核心差异总结

| 维度 | 人物 Skill | 主题 Skill |
|------|----------|----------|
| 核心 | 一个人的思维方式 | 一个领域的思维工具箱 |
| 心智模型来源 | 主要来自一个人 | 综合多人视角 |
| 表达风格 | 模拟此人的表达 | 中性但专业 |
| 矛盾处理 | 保留个人内在矛盾 | 呈现流派间分歧 |
| 验证方式 | 对比此人已知立场 | 对比领域内共识 |

### 诊断时的人物 vs 主题判断

> 来源：SKILL.md Phase 0B Step 2（第 106-108 行）

在模糊需求诊断中，花叔给出了明确的判断标准：
- 用户的需求指向某种具体的思考方式 → 人物 Skill
- 用户的需求指向某个领域的方法论 → 主题 Skill
- 不确定 → 推荐中同时包含两种类型，让用户选

## 8. 花叔的独特洞察

以下是花叔方法论中独有的、在其他 Skill 作者那里少见或没有的观点。

### 8.1 长文 > 金句

> 来源：SKILL.md "品味守则"（第 587 行）

**原话**："3000字essay比50条推文更揭示思维结构。"

**为什么独特**：主流观点倾向于"提炼精华""抓金句"，但花叔反其道而行——长文中的论证结构、转折方式、概念展开顺序，比碎片化的金句更能揭示一个人真正的思维框架。这在 Agent 1（著作）的权重设定中得到体现：书和长文是"最高"优先级，社交媒体只是"中等"。

### 8.2 争议 > 共识

> 来源：SKILL.md "品味守则"（第 588 行）

**原话**："最被争议的观点最能揭示独特性。"

**为什么独特**：常规方法论倾向于寻找"代表性观点"，花叔则认为一个人的独特性恰恰体现在他最有争议的立场上。如果一个方法论不敢触碰争议，"内在张力" section 就会空洞。这直接对应 Phase 4 的"内在张力"通过标准（至少 2 对矛盾），以及 Agent 4（他者视角）特别关注批评与争议的设计。

### 8.3 变化 > 固定

> 来源：SKILL.md "品味守则"（第 589 行）

**原话**："改变立场的地方比一直坚持的更有信息量。"

**为什么独特**：常规 Skill 倾向于把一个人的观点固化为静态模型。花叔则要求记录观点的演化——Agent 2 要提取"改变立场的瞬间"，Agent 6 要标注"思想转折点"，extraction-framework.md 专门区分"时间性矛盾"。一个人的思维进化轨迹，比他的静态观点更值得蒸馏。

### 8.4 "写不进去的那部分，才是你真正的护城河"

> 来源：SKILL.md 扉页引言（第 12 行）

**含义**：花叔承认 Skill 的边界——框架能提取，但直觉、灵感、创造力无法蒸馏。这句话既是方法论的自知之明，也是产品定位：Skill 是镜子，不是替身。对应的可操作机制是"诚实边界"——每个 Skill 必须明确列出：不能预测面对全新问题的反应、不能替代此人的创造力和直觉、公开表达与真实想法可能有差距。

### 8.5 心智模型的排他性标准

> 来源：extraction-framework.md "验证 3: 有排他性"（第 26-33 行）

**原话**："不是所有聪明人都会这样想，这个模型体现了此人的独特视角。"

**为什么独特**：大多数知识提炼方法论追求"普适性""通用性"，花叔则把"排他性"作为心智模型的判定门槛。如果所有聪明人都这么说，说明这不是此人的独特视角，不值得放进以他命名的 Skill。这是一个关键的"减法"原则——不是所有正确的观点都应该收录。

### 8.6 Agentic Protocol 从心智模型反推研究维度

> 来源：SKILL.md Phase 3 Step 2（第 481-495 行）

**为什么独特**：大多数 Skill 的角色扮演是"怎么说得像"，花叔增加了"怎么做得像"——从心智模型推导出此人面对新问题时应该搜索什么、关注什么数据。这让 Skill 从"鹦鹉学舌"升级为"可靠思维顾问"。技术实现上，每个研究维度必须来自心智模型，不能是通用的"搜索相关信息"——这是"以认知框架驱动行为"的体现。

### 8.7 检查点设计的经济学思考

> 来源：SKILL.md 检查点的成本论证

花叔在 Phase 1.5 和 Phase 2.5 两个检查点的论证中都用了成本思维："在这里拦截比在 Phase 4 返工成本低得多""提炼是主观判断最重的环节，确认后再构建，避免写完 400 行 SKILL.md 才发现方向不对"。

**为什么独特**：这不是简单的"分步确认"——花叔精确计算了每个环节的返工成本，把检查点放在成本差异最大的节点。这是一种**工程经济学的设计思维**，而非直觉式的"多确认几次"。
