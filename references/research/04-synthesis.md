# 04 — 统一工匠框架：领域共识 + 各家分歧

> 基于 01-zara-method.md / 02-huashu-method.md / 03-anthropic-method.md 的交叉提炼
> 提取日期：2026-05-09

---

## 一、领域共识（3 家都认同的 8 条原则）

### 共识 1：质量必须被验证，不能假设

三家都有独立的验证/测试体系，且都使用**外部视角**避免自评偏差。

| 来源 | 验证机制 |
|------|---------|
| Zara | 5 类强制交互元素清单、每模块至少 1 个 quiz |
| Huashu | Phase 4 三测试（已知/边缘/风格）+ 6 项通过标准 |
| Anthropic | Eval-Driven Loop + 并行对比测试 + Grading |

**统一表述**：不假设"写完了就是好的"。有独立的、可操作的验证标准，用外部视角（子 agent / grader / 用户）而非作者自评。

### 共识 2：渐进式信息披露（Progressive Disclosure）

三家都采用了分层加载的架构，不在第一时间倾泻全部信息。

| 来源 | 分层方式 |
|------|---------|
| Zara | 目录结构：CSS/JS 预制 + 模块独立 HTML + build.sh 组装 |
| Huashu | Phase 检查点：1.5 调研确认 → 2.5 提炼确认 → 3 构建，每步只加载当前需要的 |
| Anthropic | 三层加载：metadata → SKILL.md body → bundled resources |

**统一表述**：按需加载。用户/Agent 看到的永远只是当前步需要的信息，不是全部。用明确的触发条件控制加载时机。

### 共识 3：解释 Why，不只说 What

三家都拒绝机械指令，强调让执行者理解原因。

| 来源 | 表述 |
|------|------|
| Zara | 每个模块先回答"Why Should I Care"再讲"How Does It Work" |
| Huashu | "蒸馏的是 HOW they think，不是 WHAT they said" |
| Anthropic | "explain why things are important in lieu of heavy-handed MUSTs" |

**统一表述**：机械指令不可靠。让执行者理解背后的原因和原则，比堆砌 MUST/ALWAYS 更有效。

### 共识 4：独立验证防自评偏差

三家都使用子 agent 或独立评估者做验证。

| 来源 | 机制 |
|------|------|
| Zara | 子 agent 写模块，主 agent 一致性检查 |
| Huashu | Phase 4 spawn 子 agent 执行测试，非主 agent 自评 |
| Anthropic | Blind Comparison：评判者不知道输出来源 |

**统一表述**：自评不可靠。验证必须由独立于创作者的评估者执行。

### 共识 5：迭代有硬上限

三家都为迭代设置了明确的终止条件。

| 来源 | 上限 |
|------|------|
| Zara | 4-6 模块上限，不无限扩展 |
| Huashu | Phase 2→4 最多循环 2 次 |
| Anthropic | 用户满意 / feedback 全空 / 无实质进展 → 终止 |

**统一表述**：不无限打磨。设定明确的迭代上限和终止条件。过度迭代掩盖而非解决问题。

### 共识 6：产出必须自包含

三家都要求最终产物是独立、可移植的完整单元。

| 来源 | 形式 |
|------|------|
| Zara | 目录结构：打开 index.html 即用，零外部依赖 |
| Huashu | "复制整个 skill 目录就能独立使用，不依赖任何外部文件" |
| Anthropic | `.skill` 文件打包：SKILL.md + scripts + references + assets |

**统一表述**：产物独立。复制即用，不依赖外部路径、外部文件、外部服务。

### 共识 7：从用户视角设计，不从实现者视角

三家都强调从消费端出发设计接口。

| 来源 | 表述 |
|------|------|
| Zara | 课程从"学习者已知的行为"出发，不从代码结构出发 |
| Huashu | "一个意图一个入口。设计新接口时从调用者视角出发，不从实现者视角出发" |
| Anthropic | Description 覆盖用户会说的自然语言，不是技术分类 |

**统一表述**：从调用者/用户视角设计。外部接口（触发词、入口、导航）反映用户意图，不反映内部实现结构。

### 共识 8：触发精准比功能全面更重要

三家都在触发/入口设计上投入了大量精力。

| 来源 | 机制 |
|------|------|
| Zara | 多信号叠加：description + 触发短语列表 + First-Run Welcome |
| Huashu | 双入口分流 + 触发词覆盖多种表达方式（含模糊需求） |
| Anthropic | Description 作为唯一触发机制 + 对抗 undertrigger + 20-query eval |

**统一表述**：Skill 不被触发等于不存在。触发设计是 Skill 质量的入口关卡，投入和功能设计同等的精力。

## 二、各家分歧（有价值的差异）

### 分歧 1：自主执行 vs 检查点审批

| Zara | Huashu |
|------|--------|
| "Do NOT present curriculum for approval — just build it" | Phase 1.5 和 2.5 两个强制检查点，必须用户确认后才继续 |

**分析**：Zara 面向的是"用户不知道自己要什么课程"的场景——自主判断更高效。Huashu 面向的是"调研质量决定上限 + 提炼主观性强"的场景——错误成本高，检查点值得。

**适用判断**：错误成本高 → 加检查点。用户想快速得到结果 → 减少中断。

### 分歧 2：固定交互元素清单 vs 按需调整

| Zara | Huashu |
|------|--------|
| 5 类强制交互元素，每个课程必须全部出现 | 按人物/主题调整模板结构，不强制固定元素 |

**分析**：Zara 的输出类型单一（互动课程），固定元素保证最低质量标准。Huashu 的输出类型多样（人物 Skill / 主题 Skill / 更新）需灵活调整。

**适用判断**：输出类型单一 → 固定标准。输出类型多样 → 原则性指导而非固定清单。

### 分歧 3：自动化优化 vs 人工精炼

| Anthropic | Zara + Huashu |
|-----------|---------------|
| ML 方法论：train/test split + 多轮自动迭代 + test score 选最优 | 人工评审 + 双 Agent 交叉验证 + 主 Agent 裁决 |

**分析**：Anthropic 的 `run_loop.py` 把 description 优化变成自动化过程。Zara 和 Huashu 的精炼依赖人工判断和 Agent 间的交叉评审。

**适用判断**：可量化指标（触发率、token 消耗）→ 自动化。主观质量（表达 DNA、品味）→ 人工精炼。

### 分歧 4：矛盾价值 vs 一致性要求

| Huashu | Anthropic |
|--------|-----------|
| "矛盾是深度来源，不是需要修复的 Bug" | Eval 框架期望输出一致性，不确定就判失败 |

**分析**：Huashu 蒸馏的是"人"——人的内在矛盾是真实人格的体现。Anthropic 测试的是"执行质量"——不一致的输出是可修复的缺陷。

**适用判断**：对象是"人/思维" → 保留矛盾。对象是"功能/流程" → 要求一致性。

### 分歧 5：Build First vs Eval First

| Zara | Anthropic |
|------|-----------|
| 先建课程，给用户看结果再改 | 先写 test case，再写内容，循环迭代 |

**分析**：Zara 的"先建后改"降低了用户的决策负担——看结果比看计划容易判断。Anthropic 的"先评后写"防止了方向性错误。

**适用判断**：用户对最终产物有清晰预期 → Build First。产物定义本身在探索中 → Eval First。

## 三、统一工匠框架（用于 SKILL.md 构建）

基于以上共识和分歧，提取出 **8 维度 × 多级标准** 的统一框架。

### 维度 1：触发精准度

| 等级 | 标准 |
|------|------|
| ★★★ | Description 精确描述触发场景 + 覆盖多种自然语言表达 + 含模糊需求入口 + 有近失负面测试 |
| ★★☆ | Description 列出主要触发场景 + 有触发词列表 |
| ★☆☆ | Description 只描述 Skill 功能，不说明何时触发 |

### 维度 2：信息架构

| 等级 | 标准 |
|------|------|
| ★★★ | 三层加载：触发层(description) → 执行层(body) → 资源层(references/scripts)。每层独立、按需加载 |
| ★★☆ | 两层加载：触发层 + 执行层。但 body 可能过大或未分离资源 |
| ★☆☆ | 一层倾泻：所有信息堆在 body 里 |

### 维度 3：验证体系

| 等级 | 标准 |
|------|------|
| ★★★ | 独立验证（子 agent / grader / 盲比）+ 可操作通过标准 + 明确的失败信号 + 迭代上限 |
| ★★☆ | 有验证步骤但执行者与创作者同一人，或标准模糊 |
| ★☆☆ | "写好就行"，无验证环节 |

### 维度 4：用户视角设计

| 等级 | 标准 |
|------|------|
| ★★★ | 入口/触发/工作流从用户意图出发，不暴露内部实现。一个意图一个入口，不同行为用参数区分 |
| ★★☆ | 入口反映用户需求但内部结构有泄漏 |
| ★☆☆ | 按内部实现拆分入口（REST 式 list/get/create/update/delete） |

### 维度 5：诚实边界

| 等级 | 标准 |
|------|------|
| ★★★ | 3 条以上具体局限 + 标注信息截止时间 + 说明哪些场景不适用 + 承认不确定性 |
| ★★☆ | 有局限说明但空泛（"不能替代 AI"） |
| ★☆☆ | 无诚实边界，或只说优点 |

### 维度 6：产物自包含

| 等级 | 标准 |
|------|------|
| ★★★ | 单一目录/文件，复制即用。不依赖外部路径、外部文件、外部服务 |
| ★★☆ | 有外部依赖但文档化 |
| ★☆☆ | 散落在多个目录，依赖隐式路径 |

### 维度 7：表达一致性（风格约束）

| 等级 | 标准 |
|------|------|
| ★★★ | 有明确且可验证的视觉/语言风格约束（如"禁止紫色渐变""不用 Inter 字体"），每条有反例 |
| ★★☆ | 有风格指导但不可验证（"写得生动一些"） |
| ★☆☆ | 无风格约束 |

### 维度 8：迭代纪律

| 等级 | 标准 |
|------|------|
| ★★★ | 明确的迭代上限 + 终止条件 + 每轮有具体的改进维度 |
| ★★☆ | 有迭代意识但无限循环风险 |
| ★☆☆ | 一次写完不回头，或无限打磨无终止条件 |

## 四、从框架到 Skill：双模式设计

基于共识 7（从用户视角设计）和 Huashu 的双入口模式，`skill-craftsmanship-framework` 应支持两种使用模式：

### 模式 A：审计（Audit）

用户输入：一个已有 Skill / 工作流 / 产品
输出：按 8 维度逐项评分（★★★ / ★★☆ / ★☆☆）+ 最弱 2 个维度的具体改进建议

触发示例：
- "用工匠框架审计我的 XX skill"
- "检查这个工作流的 craftsmanship"
- "这个产品设计有什么 craftsmanship 问题"

### 模式 B：构建（Build）

用户输入：一个工作流或需求描述
输出：按 8 维度引导构建新 Skill，每步给出具体指导和反例

触发示例：
- "用工匠框架帮我把这个工作流变成 skill"
- "我要建一个 XX skill，按工匠标准来"
- "用 craftsmanship 标准指导我设计这个工具"

### 双模式共享的 8 维度

审计和构建使用同一套 8 维度框架，但应用方式不同：
- 审计：打分 + 找弱点
- 构建：指导 + 红绿灯检查
