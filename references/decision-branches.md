# 6 个决策分支：详细讨论 + 来源对照

> 这 5 个分歧来自 3 位 Skill 作者的方法论差异。它们不是缺陷——是不同应用场域下的最优解。理解"什么时候该听谁的"比记住"谁说了什么"更重要。

---

## 分支 1：审批策略 — 直接建还是设检查点？

### 分歧来源

| 作者 | 立场 | 原话 |
|------|------|------|
| Zara | 直接建，不审批 | "Do NOT present the curriculum for approval — just build it." (SKILL.md L105) |
| Huashu | 关键节点设检查点 | Phase 1.5 和 2.5 两个强制检查点，用户确认后继续 (SKILL.md L314-409) |
| Anthropic | 循环迭代 | "draft → test → review → improve → repeat" 直到用户满意或 feedback 全空 (SKILL.md L474-481) |

### 为什么会有分歧

Zara 的输出类型单一（HTML 互动课程），质量波动有天然上限——最多就是模块薄了、少了交互元素，方向不会全错。用户对"互动课程"有清晰的预期，看完结果容易判断好坏。

Huashu 的认知提炼主观性强，方向一旦出错就是系统性错误——"Phase 2 的方向错了，Phase 3 写的 400 行 SKILL.md 全白费"。返工成本远大于检查点成本。

Anthropic 面对的是元工具（帮助别人创建 Skill），产物类型无限多样——不存在"固定预期"，所以用循环迭代逼近。

### 判断条件

```
if 错误成本高（方向错了返工代价大）:
    → 设检查点 (Huashu)
elif 产物类型单一，用户有清晰预期:
    → 直接建 (Zara)
elif 产物定义本身在探索中:
    → 循环迭代 (Anthropic)
```

### 选错的代价

| 错误 | 后果 |
|------|------|
| 该设检查点没设 | 方向性错误导致大规模返工 |
| 不该设却设了 | 用户被频繁打断，体验差 |
| 该循环却固定 | 过早锁定方向，错过更好的方案 |

## 分支 2：输出标准 — 固定清单还是原则指导？

### 分歧来源

| 作者 | 立场 | 证据 |
|------|------|------|
| Zara | 固定 5 类强制交互元素 | 每个课程必须出现 Group Chat / Message Flow / Code-English / Quiz / Glossary Tooltip (SKILL.md L96-102) |
| Huashu | 按对象灵活调整 | 人物 Skill 有角色扮演规则 + 身份卡 + 表达 DNA；主题 Skill 去掉这些改为框架概览 + 流派对比 (SKILL.md L605-618) |

### 为什么会有分歧

Zara 的产物永远是同一种东西——互动 HTML 课程。固定清单是"同类产物的最低质量保障"，不需要每次重新定义什么是好的。

Huashu 的产物类型多样——人物 Skill、主题 Skill、更新模式——各有不同的结构需求。用同一张固定清单就是削足适履。

### 判断条件

```
if 产物类型单一（永远输出同一类东西）:
    → 固定清单 (Zara)
elif 产物类型多样（不同场景输出不同结构）:
    → 原则性指导 (Huashu)
```

### 选错的代价

| 错误 | 后果 |
|------|------|
| 固定清单用于多样产物 | 强制元素不适用当前类型，整体质量反而下降 |
| 原则指导用于单一产物 | 标准太松，每次重新发明轮子，质量不稳定 |

## 分支 3：优化方式 — 自动化还是人工精炼？

### 分歧来源

| 作者 | 立场 | 证据 |
|------|------|------|
| Anthropic | 自动化优化 | `run_loop.py`：train/test split + 多轮自动迭代 + test score 选最优 description (SKILL.md L376-394) |
| Huashu | 人工精炼 | 双 Agent 交叉评审 + 主 Agent 综合裁决 (SKILL.md L543-563) |
| Zara | 人工精炼 | 写作 agent 并行 + 主 agent 一致性检查 (SKILL.md L168-172) |

### 为什么会有分歧

触发率是可量化指标——"20 个 query，17 个成功触发 = 85%"。可量化的东西适合自动化——ML 方法论比人工更快、更稳定、可复现。

表达 DNA、隐喻质量、品味是主观的——没有客观的"隐喻质量指标"。这类问题需要多个人工视角交叉验证来逼近客观，就像论文的同行评审。

Anthropic 自己也在 grader 中承认：**主观性 skill 不应强加定量 assertion**。"Writing style, design quality, etc. — these need human judgment. Forcing quantitative assertions on them is counterproductive." (SKILL.md L202)

### 判断条件

```
if 可量化（触发率、token 消耗、通过率、速度）:
    → 自动化 (Anthropic)
elif 主观质量（表达 DNA、品味、隐喻、幽默、美感）:
    → 人工精炼 + 多视角交叉 (Huashu + Zara)
```

### 选错的代价

| 错误 | 后果 |
|------|------|
| 自动化评估主观质量 | 弱 assertion 制造虚假信心——"永远通过的 assertion 比没有更糟" |
| 人工评估可量化指标 | 慢、不稳定、每次结果可能不同 |

## 分支 4：矛盾处理 — 保留还是修复？

### 分歧来源

| 作者 | 立场 | 证据 |
|------|------|------|
| Huashu | 保留矛盾 | "矛盾是人格的核心特征，不是需要修复的 Bug。这恰恰是深度的来源。" (extraction-framework.md) |
| Anthropic | 要求一致性 | 不确定就判失败，不给 partial credit。证据必须反映实质性任务完成而非表面合规。 (grader.md L87-99) |

### 为什么会有分歧

人天然有矛盾——在不同场景用不同规则、观点随时间演变、价值观之间存在内在张力。蒸馏人时消除矛盾 = 创造一个不存在的扁平化版本。

功能和流程的 inconsistency 是缺陷——执行结果不可预测，用户体验差。这里的一致性要求是质量保障。

**注意**：这不是"Huashu 说矛盾好，Anthropic 说矛盾不好"。是"矛盾的来源和含义不同，处理方式就该不同"。

### 判断条件

```
if 评估对象是"人/思维"（人物 Skill、思维框架提取）:
    → 保留矛盾，记录为"内在张力" (Huashu)
elif 评估对象是"功能/流程"（主题 Skill、工作流 Skill、工具）:
    → 要求一致性 (Anthropic)
```

### 三类矛盾（Huashu 分类，来源：extraction-framework.md）

| 类型 | 处理方式 |
|------|---------|
| 时间性矛盾（观点演化） | 记录演化轨迹，以近期为主但提及变化 |
| 领域性矛盾（不同场景不同规则） | 分领域记录，不强求统一 |
| 本质性张力（价值观内在冲突） | 明确记录为"核心张力"——这是深度的来源 |

### 选错的代价

| 错误 | 后果 |
|------|------|
| 把人的矛盾当 bug 修 | 人格深度丢失，Skill 变成一个"完美但不真实"的纸板人 |
| 把功能的不一致当深度 | 执行结果不可预测，"上次是对的这次怎么错了" |

## 分支 5：先后顺序 — Build First 还是 Eval First？

### 分歧来源

| 作者 | 立场 | 证据 |
|------|------|------|
| Zara | Build First | 直接建课程，不给用户看大纲审批 (SKILL.md L105) |
| Anthropic | Eval First | "Capture Intent → Interview → Write test cases → Write SKILL.md → Run tests → Review" (SKILL.md L10-19) |

### 为什么会有分歧

当用户对最终产物有清晰预期时——"我知道一个互动课程长什么样"——先建出来再改是最有效的。看结果比看计划容易判断好坏。

当产物定义本身在探索中时——"我要建一个 Skill，但不确定它应该包含什么"——先写 test case 定义"成功"是防止方向性错误。

### 判断条件

```
if 用户对最终产物有清晰预期:
    → Build First (Zara)
elif 产物定义在探索中，需求不确定:
    → Eval First (Anthropic)
```

### 选错的代价

| 错误 | 后果 |
|------|------|
| Build First 用于探索性场景 | 建了一个漂亮但方向错的东西——白费功夫 |
| Eval First 用于有明确预期的场景 | 过度工程化，用户觉得流程太重 |

## 分支 6：流程保证 — 约定还是结构？

### 分歧来源

| 来源 | 立场 | 证据 |
|------|------|------|
| 2026-05-16 实战翻车 | 结构保证 | evolution-cat-infographic skill 有 Beast/Karpathy 门禁但 Agent 跳过——因为门禁结果在对话里，下游步骤无文件可检查 |

### 为什么会有分歧

这是框架自身暴露的盲区——原有的 8 维度中，维度 3（验证体系）只问"有没有门禁"，不问"门禁能否被强制执行"。对话中运行的流程和写盘的流程有本质区别：

- **对话中的流程**：Agent 按 SOP 文本顺序解读。步骤之间无结构障碍——Agent 可以从步骤 2 直跳步骤 6，系统无感知。对话压缩后中间状态全部丢失。
- **写盘的流程**：每步产出文件，下游验证上游文件存在。步骤之间有结构障碍——缺文件就报错，不可能"悄悄地跳过"。

### 判断条件

```
if 流程有 ≥3 个串行步骤，且中间步骤的产出影响最终质量:
    → 结构保证。每个步骤产出文件，下游验证上游
elif 流程 ≤2 步，或中间状态无复用价值:
    → 约定保证足够
```

### 选错的代价

| 错误 | 后果 |
|------|------|
| 用约定保证需要文件链的流程 | Agent 跳过关键步骤，下游无感知。本次翻车：SOP 写了创意会但因无文件化门禁 + 无下游验证，Agent 从搜索直冲 HTML 生成 |
| 用结构保证不需要文件链的流程 | 过度工程，简单任务被文件读写拖慢 |

### 结构保证的最小实现

不需要复杂的框架——三件事就够了：

1. **每个步骤声明输出文件**（如 `_work/{选题}/内容策划.md`）
2. **下游步骤第一行动作：检查输入文件是否存在**（如 `if not exists 内容策划.md → 硬阻断`）
3. **门禁结果写盘**（如 `_work/{选题}/创意会门禁.md`，含 PASS/FAIL 判词）

这是对已有 SOP 的最小侵入式改造——不改流程逻辑，只在步骤间插入文件存在性检查。

### 本次翻车的具体修复

evolution-cat-infographic 的 production-sop.md 步骤 5（前置检查）第一行动作：

```
>> 检查 _work/{选题}/内容策划.md 是否存在 → 不存在则硬阻断："步骤 3 未完成"
>> 检查 _work/{选题}/创意会门禁.md 是否存在且 PASS → 不存在/FAIL 则硬阻断
```

这两行就是结构保证和约定保证的分界线。没有这两行，SOP 里的 ⏸ 和门禁描述都是建议。

## 总结：分歧的使用原则

1. **不分对错**——每个分歧的两端都是在特定场景下的最优解
2. **给判断条件，不给唯一答案**——框架的价值在于帮使用者选，不是替使用者选
3. **标注选错代价**——知道"选错了会怎样"和知道"该选哪个"同样重要
4. **分歧本身是最有价值的教学内容**——它揭示了 Skill 设计的深层权衡
5. **分支 6 是框架自噬的产物**——它来自框架自身审计失败。如果一个声称审计 Skill 的框架自己漏掉了"门禁无法强制执行"，这个盲区必须变成框架的一部分
