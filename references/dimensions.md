## 维度 1：触发精准度（含触发互斥）

**为什么重要**：Skill 不被触发等于不存在。Anthropic 发现 Claude 有 undertrigger 倾向——description 需要写得 "pushy"，主动列出各种触发场景。

### 详细标准

**★★★（触发精准 + 互斥声明）**
- Description 包含：Skill 做什么 + 具体触发场景 + 多种自然语言表达（含非正式/模糊说法）
- 对抗 undertrigger：主动列出"即使用户没有明确说 X，只要意图是 Y 也应该触发"
- 有近失负面测试：知道什么场景不该触发，能区分相似但不同的需求
- 触发词覆盖：正式说法 + 随意口语 + 含错别字的输入
- **触发互斥声明**：当存在同域兄弟 Skill 时，明确声明"这不归我管，应触发 X"。互斥声明包含具体的错误触发示例和正确的路由目标

**★★☆（触发基本覆盖）**
- Description 列出了主要触发场景
- 但缺少边缘覆盖——如果用户换一种说法、或表达模糊，就可能触发不了
- 没有针对 undertrigger 的主动策略
- **缺少触发互斥声明**：同域有兄弟 Skill 但未声明边界

**★☆☆（触发不明确）**
- Description 只描述 Skill 功能（"这个 Skill 用于 XXX"）
- 用户必须用精确的关键词才能触发
- 模糊表达、非正式说法、或近义需求无法触发

### 触发互斥专项检查（审计时必问）

当审计对象是工作流 Skill 且存在同域兄弟 Skill 时，必须检查以下 3 个问题：

| # | 检查项 | 不通过信号 |
|---|--------|----------|
| 1 | 是否存在同域兄弟 Skill 可能被误触发？ | Skill A 和 Skill B 都处理"写作"类关键词 |
| 2 | Description 是否声明了"不归我管"的场景？ | 负面测试仅覆盖"完全不相关的场景"，不覆盖"兄弟 Skill 的场景" |
| 3 | 互斥声明是否包含正确的路由目标？ | 只说了"不是我"但没说是谁 |

**本次翻车**：evolution-cat-article 和 evolution-cat-infographic 都处理内容生产。用户说"图文"→ Agent 调了 article。两个 Skill 的 description 都没有声明"如果用户说图文，不要触发我，去触发 infographic"。

### 常见失败模式

| 模式 | 示例 | 问题 |
|------|------|------|
| 功能描述式 | "An interactive course generator from code" | 用户说"教我这个代码库"时可能不触发——"course"和"教"之间的语义鸿沟 |
| 过度特化式 | "Turn a GitHub repo into an HTML course" | 本地文件夹、当前项目无法触发 |
| 缺失负面边界 | 不区分"这个 Skill 该做的"和"看起来像但不是它做的" | 在错误场景触发，浪费用户时间 |
| **同域无互斥** [2026-05-16] | article 和 infographic 都响应"写作""图文" | 用户说"图文"→ 随机选一个，选错率随 Skill 数增长 |

### 修复示例

**Before (★☆☆)**：
```
description: A skill for auditing quality.
```

**After (★★★)**：
```
description: |
  Skill quality audit and build framework. Use when the user wants to:
  check/evaluate/review the quality of a skill/workflow/product ("检查这个skill",
  "这个工作流有什么问题", "评估质量"), improve/upgrade an existing skill
  ("怎么改进", "帮我优化"), or build a new skill from scratch with craftsmanship
  standards ("帮我把这个流程变成skill", "按工匠标准建一个").

  Also trigger when the user asks abstract questions about skill design quality,
  even without naming a specific skill — e.g. "怎么判断一个skill好不好",
  "skill设计有什么原则".
```

### 触发互斥声明示例

**有互斥声明**（★★★）：
```
description: |
  进化猫小绿书图文生产。用于将选题转化为 1080×1440 四页卡片。
  
  **触发边界**：
  - 用户说"图文""小绿书""信息图""卡片"→ 触发本 Skill
  - 用户说"文章""长文""公众号""写一篇"→ 不触发本 Skill，应触发 evolution-cat-article
  - 用户说"写作方向""研究写作"→ 追问一句：图文还是文章？不要替用户决定
```

### 触发互斥在审计中的权重

触发互斥不是"有更好"——同域 Skill 缺少互斥声明直接拉低维度 1 至少一级。如果 2+ 次实战出现误触发，则直判 ★☆☆。

## 维度 2：信息架构

**为什么重要**：上下文窗口是有限资源。一次加载 2000 行的 SKILL.md，其中 1500 行在当前任务中用不上，浪费的不只是 token——是模型的注意力。

### 详细标准

**★★★（三层加载）**
- L1 → description（始终在上下文，~100 词）
- L2 → SKILL.md body（触发时加载，< 500 行）
- L3 → references/scripts/assets（按需加载，脚本可执行不加载）
- Body 接近 500 行时主动拆分，给出清晰的跳转指引
- 大 reference 文件（>300 行）有目录

**★★☆（两层加载）**
- 有 description + body 的分离
- 但 body 过大（500+ 行）且未拆分
- 或 resources 存在但未在 body 中说明何时读取

**★☆☆（一层倾泻）**
- 所有信息在 body 中
- 无 references/ 目录或 scripts/ 目录
- 上下文窗口被不必要的内容占据

### 常见失败模式

| 模式 | 问题 | 修复 |
|------|------|------|
| Body 膨胀 | SKILL.md 超过 800 行 | 拆分：body 保留流程，细节进 references/ |
| 无跳转指引 | Body 说"详见 X"但 X 不在 skill 目录内 | 所有引用指向 skill 内部文件 |
| 脚本加载到上下文 | 在 body 中粘贴完整脚本 | 放 scripts/ 目录，body 只写调用方式 |
| 大文件无目录 | 300+ 行 reference 无导航 | 顶部加目录 |

## 维度 3：验证体系

**为什么重要**：自评偏差是真实存在的。创作者会忽略自己最薄弱的地方——因为那正是他的盲区。独立验证是唯一的校正机制。

### 详细标准

**★★★（独立验证）**
- 验证由独立于创作者的评估者执行（子 agent / grader / 盲比）
- 有可操作的通过标准（能判定 PASS/FAIL，没有"看情况"）
- 每个标准有明确的失败信号
- 有迭代上限和终止条件
- 测试覆盖：正常路径 + 边缘情况 + 应失败的情况

**★★☆（有验证但不独立）**
- 有测试/验证步骤
- 但执行者与创作者同一人（自评）
- 或通过标准模糊（"应该是高质量的"）

**★☆☆（无验证）**
- "写完就完了"
- 无测试、无检查清单、无外部评估

### 常见失败模式

| 模式 | 问题 |
|------|------|
| 自评 | 创作者验证自己的产出——盲区是藏在自己看不见的地方 |
| 标准模糊 | "表达应该自然"——什么是自然？无法判定 PASS/FAIL |
| 无边缘测试 | 只测正常路径，不测边界 |
| 无限迭代 | 没有"够了"的标准，一直改 |
| 弱 Assertion | 永远通过的检查项——制造虚假信心，比没有更糟 |

### Anthropic 的 Assertion 黄金规则（适用时参考）

一条好的 assertion：
1. 可客观验证（不需要主观判断）
2. 有区分力（skill 成功时通过，失败时不通过）
3. 名字描述性强（扫一眼就知道检查什么）

一条坏的 assertion：永远通过，或需要主观判断，或名字像 `check_1`。

## 维度 4：用户视角设计

**为什么重要**：REST 式细粒度拆分对 AI Agent 是反模式——增加 token 开销和选择错误率。Agent 不是浏览器，它需要"我要做 X"对应一个明确的入口。

### 详细标准

**★★★（用户意图驱动）**
- 入口以用户意图为拆分单位（"我要做 X"），不以内部实现为拆分单位
- 不同行为用参数区分，不新开入口
- 调用者不需要知道内部用了几个模块、几个步骤
- 触发词覆盖用户会说的自然语言（从调用者视角出发）

**★★☆（有用户意识但结构泄漏）**
- 入口大体反映用户需求
- 但用户需要了解内部结构才能正确使用（"先调 A 再调 B"不该是用户需要知道的）
- 或入口拆分与内部模块一一对应

**★☆☆（实现驱动）**
- REST 式 CRUD 拆分：list/get/create/update/delete 各一个入口
- 用户需要理解内部模块划分才能选对入口
- 新增一个功能就加一个新入口（入口膨胀）

### 常见失败模式

| 模式 | 示例 | 问题 |
|------|------|------|
| CRUD 碎片化 | `skill-audit`, `skill-build`, `skill-check` 三个独立 Skill | 用户要理解三个 Skill 的边界和关系 |
| 实现泄漏 | 入口叫 `phase1-research` | 暴露了内部流程阶段 |

### 花叔的"一个意图一个入口"原则（来源：CLAUDE.md L39-43）

> 以 agent 的意图为拆分单位，不以内部实现为拆分单位。不同行为用参数区分，不新开入口。REST 式 CRUD 细粒度拆分是反模式——增加 token 开销和选择错误率。

**适用范围**：工具、Skill、MCP、API、CLI、文件接口——凡 agent 消费的东西。

## 维度 5：诚实边界

**为什么重要**："一个不告诉你局限在哪的 Skill，不值得信任"（花叔）。诚实边界是可信度的基准线——不是减分项，是加分项。

### 详细标准

**★★★（诚实且具体）**
- 3 条以上具体局限，每条可独立验证
- 标注信息截止时间（调研/数据的时间点）
- 明确说明哪些场景不适用（"这个 Skill 不能用于 XX"）
- 承认不确定性和推测部分
- 局限是具体可操作的（不是"不能替代 AI"这种空话）

**★★☆（有局限但空泛）**
- 有 1-2 条局限说明
- 但内容空泛——"不能替代本人""仅供参考"
- 缺少具体的场景限制和信息截止时间

**★☆☆（无诚实边界）**
- 只介绍 Skill 能做什么
- 不说不适用场景
- 不说数据截止时间
- 不承认推测 vs 事实

### 常见失败模式

| 模式 | 示例 | 为什么不够 |
|------|------|-----------|
| 空话式 | "本 Skill 不能替代本人" | 对使用者没有任何实际指导意义 |
| 免责式 | "仅供参考，不构成建议" | 法律免责不是诚实边界 |
| 隐藏式 | "可能有一些局限" | 不说具体是什么——等于没说 |

### 好的诚实边界示例

```
## 诚实边界

- 调研截止 2026-05-09。此日期之后的新公开发言未纳入
- 公开表达与私下想法可能有差距——本 Skill 仅基于可获取的公开信息
- 不能预测面对全新类型问题的反应——超出此人已展示过思维模式的领域
- 心智模型 2 和 3 基于有限信息推测（来源 < 5 条），标注为"推测"
- 负面评价和争议已收录，但批评者视角的信息可能不如正面信息充分
```

## 维度 6：产物自包含

**为什么重要**：你不可能在 Skill 里写"参考 ~/Documents/some-notes.md"——那是你的环境，别人的环境里没有这个文件。开源分发的前提是自包含。纯个人使用的 Skill 可放宽至 ★★☆（依赖文档化即可）。

### 详细标准

**★★★（完全自包含）**
- 单一目录，复制到任何位置可独立使用
- 不依赖外部路径（~/Documents/、../../../other-project/）
- 所有引用文件在 Skill 目录内部
- 所需工具在 scripts/ 内或为系统标准工具（bash/python3/git）
- 如有外部依赖（如 API Key），在 description 或 body 中明确说明

**★★☆（有文档化的外部依赖）**
- 有外部依赖但文档化（"需要先安装 XX"）
- 或引用路径相对但跨项目

**★☆☆（隐式依赖）**
- 引用外部文件路径（"参考 ~/Documents/notes.md"）
- 依赖全局安装的工具（"用我的 custom-tool"）
- 散落多个目录，无法一次性复制

### 花叔的自包含标准（来源：SKILL.md L169）

> "所有调研文件必须存在 skill 目录内部。Skill 必须是自包含的——复制整个 skill 目录就能独立使用，不依赖任何外部文件。这是为开源分发设计的核心原则。"

## 维度 7：表达一致性

**为什么重要**：不可验证的风格约束 = 没有约束。"写得生动一些"对 AI 没有信息量——必须具体到能判定 PASS/FAIL。

### 详细标准

**★★★（可验证的风格约束）**
- 有具体的、可 PASS/FAIL 判定的风格规则
- 每条规则有反例（"不要这样 → 像这样"）
- 覆盖视觉风格（如有 UI 输出）或语言风格（如有文本输出）
- 禁止项明确，不含糊

**★★☆（有指导但不精确）**
- 有风格方向（"保持专业""简洁""生动"）
- 但不可验证——两个评审员可能给出不同判定
- 缺少反例

**★☆☆（无风格约束）**
- 每次输出风格随机
- 或只依赖 AI 的默认风格

### 常见失败模式

| 模式 | 示例 | 问题 |
|------|------|------|
| 形容词式 | "写得生动一些" | 什么是生动？无法判定 |
| 无反例 | "使用专业语气" | 给出一个"不专业的例子"对比才有效 |
| 过于宽松 | "根据场景调整风格" | AI 无法判断哪个场景用哪种风格 |

### Zara 的风格约束示例（来源：code-explainer）

好的风格约束：
- ✅ "禁用 Inter、Roboto、Arial、Space Grotesk 做标题字体"
- ✅ "代码块禁止水平滚动条"
- ✅ "每屏至少 50% 视觉面积"
- ✅ "用暖色 off-white 背景，禁用冷白和蓝色调"
- ✅ "测验考应用不考记忆——最好的是'What would you do?'场景题"

每一条都能明确判定 PASS/FAIL。

## 维度 8：迭代纪律

**为什么重要**：无限打磨是在掩盖信息源问题——不是你写的不够好，是你手里的素材就这么多。2 轮后仍有不达标项 → 标注在诚实边界里，交付当前最优版本。

### 详细标准

**★★★（有纪律的迭代）**
- 明确的迭代上限（如"最多 2 轮"或"最多 N 个循环"）
- 明确的终止条件（不依赖"看起来好了"这种主观判断）
- 每轮有具体的改进维度（不是"再改改"）
- 超过上限后：标注薄弱维度在诚实边界中，交付当前最优版本

**★★☆（有迭代但无纪律）**
- 会迭代改进
- 但无明确的轮次上限或终止条件
- 可能无限打磨，或一次就停

**★☆☆（无迭代）**
- 一次写完不回头
- 或无限打磨无终止条件

### 常见失败模式

| 模式 | 问题 | 修复 |
|------|------|------|
| 完美主义陷阱 | 一直改，永远不交付 | 设轮次上限 + 超过上限标注诚实边界 |
| 一次性综合症 | 写完不管 | 至少设一轮 review |
| 无方向的迭代 | 每次改但不知道在改进什么 | 每轮聚焦 1-2 个维度 |

### 三家迭代纪律对比

| 来源 | 上限 | 终止条件 |
|------|------|---------|
| Zara | 4-6 模块 | 模块数封顶，不无限扩展 |
| Huashu | Phase 2→4 最多 2 轮 | 2 轮后仍有不通过 → 诚实边界标注 |
| Anthropic | 最多 5 轮 description 优化 | 用户满意 / feedback 全空 / 无实质进展 |

## 维度 9：交付物链

**为什么重要**：无交付物链的 SOP 是一张建议清单——Agent 可以跳过任何步骤，因为下游不会因缺少上游文件而失败。infographic 翻车：SOP 写了创意会但无文件产出，Agent 从步骤 2 直冲步骤 6 无结构性阻拦。

SOP 步骤之间的文件依赖链。每一步的输入是上一步的输出文件。

### 详细标准

**★★★（文件链驱动）**
- 每个 SOP 步骤声明：输入文件（上一步产出）和输出文件（本步骤产出）
- 下游步骤执行前必须验证上游文件存在。缺失 → 硬阻断并报错，错误信息命名缺失文件
- 步骤之间的文件依赖形成完整链，从第一步到最后一步
- 步骤依赖文件不存在时，Agent 不能自行判断"差不多可以继续"——必须停
- 人审暂停点也对应文件存在性检查（如"内容策划.md 存在 → 可进入人审"）

**★★☆（有文件产出但无验证）**
- 步骤产出文件，但下游步骤不检查上游文件是否存在
- 依赖 Agent 按顺序执行（记忆/自觉），而非结构保证
- 缺失上游文件时能继续执行，直到后续步骤出错才发现

**★☆☆（无文件级依赖）**
- 步骤仅在 SOP 中以文本描述存在
- 步骤间无任何文件传递
- 流程执行完全依赖 Agent 按文本顺序解读

### 常见失败模式

| 模式 | 问题 | 修复 |
|------|------|------|
| 文本链 | SOP 写了"步骤 3 完成后进入步骤 4"，但步骤 4 不检查步骤 3 的产出文件 | 步骤 4 第一行动作：检查步骤 3 的产出文件是否存在 |
| 隐式依赖 | 步骤 5 需要"内容策划的结果"但未指定是哪个文件 | 明确文件名：`_work/{选题}/内容策划.md` |
| 人审无锚点 | SOP 说"⏸ 停等人审"但无文件标记人审状态 | 人审通过后写 `人审通过.md` 或更新门禁文件状态 |
| 跳过无感知 | Agent 跳过了步骤 3，步骤 5 照样执行——系统无任何报错 | 步骤 5 第一行：`检查 内容策划.md 和 创意会门禁.md 存在且 PASS` |

### 审计检查清单

对每个 SOP 步骤，逐项检查：

| # | 检查项 | 不通过信号 |
|---|--------|----------|
| 1 | 步骤 N 是否声明了输入文件？ | "基于上一步的分析"——但没说分析在哪 |
| 2 | 步骤 N 是否声明了输出文件？ | 产出留在对话中，未写盘 |
| 3 | 步骤 N+1 是否验证了步骤 N 的输出文件存在？ | 步骤 N+1 直接从对话中取信息 |
| 4 | 全流程的文件链是否连续（无断点）？ | 某两步之间无文件传递 |

### 交付物链模板（供 Skill 设计参考）

```
步骤 1: 选题分类
  输入：用户需求
  输出：_work/{选题}/前置检查.md（含分类结果、布局模式锁定）

步骤 2: 素材抓取
  输入：_work/{选题}/前置检查.md（确认分类后才抓素材）
  输出：_work/{选题}/搜索/素材.md
```

## 维度 10：层层外化门禁

**为什么重要**：对话上下文是易失的——压缩后门禁结果消失，Agent 记不清门禁过了没过。写盘 = 可追溯 = 可强制执行。infographic 翻车：门禁定义了完整流程但结果留在对话里，下游步骤无文件可检查。

质量门禁必须写盘。在对话中运行的"门禁"不是门禁——是建议。

### 详细标准

**★★★（门禁文件化）**
- 每个质量门禁产出可检查的文件记录（如 `创意会门禁.md`）
- 门禁文件包含：门禁名称、检查了什么、判词（PASS/FAIL）、具体失败项
- 下游步骤的第一行动作：检查门禁文件是否存在且 PASS
- 门禁文件不存在或显示 FAIL → 下游步骤拒绝执行
- 门禁文件是人类可读的——人审时可以打开看 Agent 在门禁里做了什么
- 门禁产出在对话压缩后仍然存在（不受上下文窗口影响）

**★★☆（门禁存在但未文件化）**
- SOP 中描述了门禁流程（如 "Beast 标题关 → Karpathy 诚实关"）
- 但门禁结果留在对话中，无文件产出
- 门禁执行与否取决于 Agent 自觉——可被跳过而系统无感知
- 对话压缩后门禁结果丢失

**★☆☆（无质量门禁）**
- 无质量门禁，或仅以"应该检查质量"的模糊建议形式存在
- 无具体检查项，无 PASS/FAIL 判定标准

### 常见失败模式

| 模式 | 问题 | 修复 |
|------|------|------|
| 对话门禁 | 门禁全在对话里跑，无文件产出。Agent 可以"在脑子里过一遍"然后声称过了 | 门禁结果必须写成文件。不说"我检查了"——说"门禁结果在 X.md 里" |
| 门禁无判词 | 门禁文件只写了"通过"但没有具体检查了哪些项 | 门禁文件必须列出每项检查 + 结果（PASS/FAIL）+ 失败项的具体问题 |
| 下游不检查 | 门禁文件写了，但下游步骤不读 | 下游步骤第一行动作：检查门禁文件存在且 PASS |
| 人审无法参与 | 门禁结果在对话里，人看不到 Agent 在门禁里做了什么 | 门禁文件是人类可读的文本/Markdown |

### 外化门禁模板（供 Skill 设计参考）

```markdown

**门禁类型**：Beast 标题关 + 传播关 → Karpathy 诚实关 + 认知深度关

| 检查项 | 结果 | 说明 |
|--------|------|------|
```

### 外化门禁在审计中的权重

门禁无文件化产出的 Skill，无论其他维度多强，维度 10 不得高于 ★★☆。两次以上实战出现"门禁被跳过"的，直判 ★☆☆。

## 附录：主题 Skill vs 人物 Skill 在审计中的差异

审计时应根据 Skill 类型调整某些维度的判定标准：

| 维度 | 人物 Skill | 主题 Skill |
|------|----------|----------|
| **表达一致性** | 需评估表达 DNA 辨识度（读者的感受："删掉名字还能认出是谁"） | 需评估表达是否专业中性（读者的感受："不是某个人在说话，是这个领域的框架在说话"） |
| **诚实边界** | 需包含：公开表达 vs 真实想法差距、不能替代直觉/创造力 | 需包含：框架适用的边界条件、流派的局限、各家的盲区 |
| **验证体系** | 边缘测试：用此人没讨论过的问题测试（"基于模型推断，可能..."） | 对比测试：对比领域内公认的经典案例和共识 |
| **信息架构** | 包含身份卡、角色扮演规则、表达 DNA | 去掉身份卡和角色扮演，改为框架概览 + 流派对比 |
